# 阶段 1 统一术语

本文是阶段 1 及后续阶段的规范词汇表。中文为解释，英文标识为跨文档稳定用词；实现字段是否采用这些名称由后续阶段决定。

## 参与者与范围

| 术语 | 统一定义 |
|---|---|
| customer（顾客） | 提出菜单查询或订餐请求的人；模拟环境中的顾客均为虚构角色。 |
| caller（来电者） | 通过未来电话入口发起会话的人；不当然等同于付款人、收餐人或 customer。阶段 1 没有真实来电。 |
| restaurant（餐厅） | 提供菜单并对订单作最终接受或拒绝决定的经营主体；本阶段仅指虚构模拟餐厅。 |
| branch（门店） | restaurant 下实际履约的单一地点。订单只能绑定一个明确 branch。 |
| operator（运营人员） | 负责菜单、营业信息和服务配置的餐厅侧人员，不等同于 human agent。 |
| human agent（人工客服） | 接收高风险、异常或顾客主动请求的会话并决定后续操作的人。阶段 1 只定义政策，不提供坐席。 |
| restaurant administrator（餐厅管理员） | 未来管理门店人员权限和餐厅配置的角色。 |
| system administrator（系统管理员） | 未来负责技术配置、审计与事件响应的角色；不得默认查看完整顾客内容。 |
| tester（测试人员） | 使用合成数据执行验证的人；无权使用真实顾客数据填充测试集。 |

## 会话与订单

| 术语 | 统一定义 |
|---|---|
| conversation（对话） | 顾客与系统或人工围绕一次服务目的交换消息的逻辑记录，可跨多个技术连接。 |
| call（通话） | 未来电话平台上的一次媒体连接；连接中断不自动结束 conversation。阶段 1 无真实 call。 |
| session（会话） | 隔离上下文与草稿状态的技术边界。任何订单操作必须绑定且只能影响一个 session。 |
| draft order（订单草稿） | 尚未获得有效最终确认、可逆且未产生商家侧订单的当前订单内容。 |
| customer confirmation（顾客确认） | 顾客在听到或看到最终内容、数量、总价、履约方式及重要字段后作出的明确同意。草稿重要字段变化后旧确认失效。 |
| order submission（订单提交） | 在有效 customer confirmation 后，将订单请求发送给指定餐厅或其权威订单系统的动作。发送开始或完成不代表接受。 |
| merchant acceptance（商家接受） | 指定餐厅或其权威订单系统明确返回订单已接受的结果。unknown、pending、timeout 和本地订单号均不是接受。 |
| successful order（成功订单） | 已取得可归因、可审计的 merchant acceptance 的订单；阶段 1 只定义该标准，不产生成功订单。 |
| completed（履约完成） | 餐厅完成制作及约定交付后的业务事件，不等同于 successful order。 |
| handoff（人工接管） | 自动流程停止目标操作，保留草稿并把最少必要摘要交给 human agent 的过程。发起接管不代表已连接人工。 |
| refusal（拒绝处理） | 系统停止不允许的目标操作，并给出简短、安全且不泄露内部信息的说明。 |
| escalation（升级） | 因风险、故障或权限需要，把事项提交给更高权限人员或事件流程；可发生在 handoff 前后。 |

## 食品安全

| 术语 | 统一定义 |
|---|---|
| allergy（过敏） | 顾客声明可能触发免疫反应的食品安全信息；不得降级为普通偏好或普通备注。 |
| severe allergy（严重过敏） | 顾客明确说明严重、可致命、曾有严重反应，或描述呼吸困难等高风险信息的 allergy；必须停止自动完成订单并转人工。 |
| intolerance（不耐受） | 顾客声明的非过敏性食物不耐受。仍需准确记录和确认，但不得擅自改写为 allergy。 |
| cross-contamination（交叉污染） | 过敏原通过器具、表面、人员或制作流程非预期进入食品的可能性。系统不得保证不存在交叉污染。 |
| dietary preference（饮食偏好/普通忌口） | 不吃、少放或口味偏好，但顾客未声明为 allergy 或 intolerance。不得与二者混用。 |

## 数据与环境

| 术语 | 统一定义 |
|---|---|
| personal data（个人资料） | 可直接或间接关联可识别个人的数据，例如电话、地址和顾客姓名；具体法律定性待适用法律审查。 |
| sensitive operational data（敏感运营数据） | 泄露后可造成安全或业务损害的内部数据，例如风险规则、商家接口状态和审计细节。 |
| synthetic data（合成数据） | 为测试人工构造、且不对应真实个人、真实餐厅或真实交易的数据。 |
| transcript（转写） | 由语音或人工记录得到的文字；可能包含 personal data，不能因其是文本就降低保护等级。 |
| raw audio（原始音频） | 未经转写的声音记录；可能包含身份、环境及健康相关信息，按 HIGH_SENSITIVITY 处理。 |
| audit event（审计事件） | 为追踪关键决定而记录的结构化事件；应记录原因代码和状态变化，不应默认复制完整顾客原话。 |
| simulation（模拟） | 使用虚构餐厅、合成顾客和非真实外部接口验证设计的环境，不提供真实服务。 |
| production（生产） | 面向真实顾客和商家的受控服务环境；需要完成安全、运营、法律及餐厅验证后才可启用。 |

## 必须保持的语义分离

```text
顾客确认（customer confirmation）
≠ 已发送到餐厅（order submission）
≠ 餐厅已接受（merchant acceptance）
≠ 订单制作或交付完成（completed）
```

当前仓库的 `submitted` 和 `MOCK-ORDER-0001` 只表示本机演示的 mock 提交状态，不能映射为 merchant acceptance 或 successful order。
