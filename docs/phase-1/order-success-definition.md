# 阶段 1 订单成功定义

本文定义未来订单生命周期的产品语义，不新增或修改当前状态机。当前仓库的 `submitted=True` 和固定 mock 订单号只代表本机演示提交，不能证明餐厅收到或接受订单。

## 1. 概念事件

| 事件 | 严格定义 | 可对顾客表达的最高语义 |
|---|---|---|
| `draft_created` | 当前 session 中出现至少一个草稿项 | “已加入草稿” |
| `customer_confirmed` | 顾客明确确认了绑定版本的完整订单与重要字段 | “已收到你的确认” |
| `submission_started` | 系统在有效确认后开始向指定商家端发送 | “正在发送给餐厅” |
| `merchant_pending` | 商家端明确表示处理中，或系统仍在等待权威结果 | “餐厅仍在处理，尚未接受” |
| `merchant_accepted` | 指定餐厅或权威订单系统返回明确、可归因、可审计的接受结果 | “餐厅已经接受你的订单” |
| `merchant_rejected` | 权威商家端明确拒绝 | “餐厅暂未接受这笔订单” |
| `submission_failed` | 发送失败、结果无法验证、超时且查询后仍未知，或安全校验失败 | “尚未确认餐厅接受” |
| `cancelled` | 顾客或授权人员在允许的阶段终止订单；是否已被商家接受必须同时说明 | “已取消请求/正在确认取消结果” |
| `completed` | 餐厅记录制作和约定交付完成 | “订单已完成” |

事件之间不能以名称相近为由合并。尤其 `customer_confirmed` 不是 `submission_started`，`submission_started` 不是 `merchant_accepted`，`merchant_accepted` 也不是 `completed`。

## 2. 成功与非成功

只有餐厅或其权威订单系统明确返回接受结果后，订单才能称为“餐厅已接受”。一个 successful order 必须同时具备：

1. 唯一 session、restaurant、branch、draft version 和 idempotency key；
2. 对该版本有效的 customer confirmation；
3. 经过授权接口取得的明确 `merchant_accepted`；
4. 结果可归因到同一订单请求，且审计记录不包含不必要个人资料；
5. 没有尚未解决的严重过敏、跨餐厅或商家身份异常。

下列情况均不得称为成功或商家已接受：

- 顾客只说“对”“好”或其他未绑定复述的确认词；
- 系统生成本地订单号或保存到内存/数据库；
- 请求已发送或网络连接建立；
- 网络请求超时或重试中；
- POS/商家接口返回 `unknown`、`pending`、空结果或无法验签的回调；
- 人工尚未确认或转接仍在进行；
- 商家回调未到达、无法关联或来自错误门店；
- 当前仓库返回 `MOCK-ORDER-0001`。

## 3. 确认、提交和接受协议

有效 customer confirmation 必须在最终复述后发生，明确覆盖菜品、数量、总价、履约方式以及适用的地址、电话、辣度、重要备注和系统推断。确认记录必须绑定当前草稿版本；问句、沉默、噪声、旧对话或另一 session 的回答均无效。

确认后只允许进入 `submission_started`。未来商家接口的 `accepted` 必须有权威来源、明确语义和幂等关联；不能从 HTTP 成功码、本地生成 ID 或无错误异常推断接受。超时后应查询权威状态；仍未知则保持 `merchant_pending` 或进入 `submission_failed`/`HANDOFF`，绝不猜测成功。

## 4. 推荐话术语义

本阶段只定义语义，不实现普通话、粤语或英语话术：

- 顾客确认后：“已收到你的确认，我正在把订单发送给餐厅。”
- 商家明确接受后：“餐厅已经接受你的订单。”
- 商家拒绝或失败时：“餐厅暂未接受这笔订单，需要重新选择或转人工处理。”
- 结果未知时：“我还没有取得餐厅接受结果，不能把这笔订单说成成功。”

## 5. 不变量

| invariant_id | 不变量 | 违反后的默认处理 |
|---|---|---|
| INV-001 | 未确认不得提交 | 阻止提交，回到 `CONFIRM` |
| INV-002 | 同一有效确认不得生成多个外部订单 | 幂等查询并 `HANDOFF` |
| INV-003 | 不得跨 session、restaurant 或 branch 提交 | `REFUSE`/安全升级，不应用状态 |
| INV-004 | `unknown`、`pending`、timeout 不得当作 accepted | 保持未成功并查询或 `HANDOFF` |
| INV-005 | 商家拒绝后不得继续显示成功 | 修正状态、告知并 `HANDOFF` |
| INV-006 | 顾客修改订单后旧确认失效 | 增加草稿版本并重新 `CONFIRM` |
| INV-007 | 价格、履约、地址、电话、数量或重要备注变化后必须重新确认 | 阻止提交并重新复述 |
| INV-008 | 严重过敏或未核实交叉污染不得自动进入成功 | `HANDOFF`，禁止安全保证 |
| INV-009 | 商家回调必须关联原请求且防重放 | 拒绝无效回调并记录审计事件 |
| INV-010 | 转人工失败不得自动提交 | 保留草稿并明确告知失败 |
| INV-011 | 订单号存在不等于接受 | 按权威商家状态显示 |
| INV-012 | completed 不能由系统根据时间自行推断 | 只接受权威履约事件 |

## 6. 当前实现映射

| 当前字段/行为 | 阶段 1 解释 | 后续要求 |
|---|---|---|
| `current_order` | 内存 draft order | 保持 session 与门店隔离 |
| `submitted` | 演示锁定标志 | 不直接映射为任何商家成功事件 |
| `submitted_order_id` | 固定 mock ID | 不作为 merchant acceptance 证据 |
| `ConfirmationAgent.submit_order()` | 内存 mock 行为 | 阶段 1 不修改；未来提交必须遵守确认和幂等协议 |
| 已提交后锁定 | 有用的演示不变量 | 未来需区分商家待定、接受、拒绝和失败 |

阶段 1 不新增上述概念事件的生产状态机；后续阶段实现时必须以本定义为验收依据。
