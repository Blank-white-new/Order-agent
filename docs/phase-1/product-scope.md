# 阶段 1 产品范围

状态：阶段 1 规范。本文定义未来产品边界，不实现数据库、三语解析、电话、POS、支付、后台或生产业务代码。处理规则使用稳定需求 ID `REQ-001` 至 `REQ-030`。

## 1. 产品目标与当前基线

产品方向是帮助普通餐饮门店处理电话订餐中的菜单问答、草稿编辑、信息确认、向商家提交及异常转接，减少重复人工操作，同时保留食品安全、争议处理和失败场景的人工控制。

阶段 1 只建立产品、安全和验收规范。当前仓库实际是规则优先的中文本机演示：Orchestrator 是唯一入口，菜单、价格和配送费来自服务层，会话与 mock order 仅存在内存，语音为可选本机中文演示，LLM fallback 默认离线。当前没有商家接受、真实订单、真实支付、真实电话、真人接管、生产三语或 POS 能力。

## 2. 市场、餐厅和语言边界

- `REQ-001`：任何文本或未来语音入口必须由 Orchestrator 统一裁决；子 agent 不得直接修改状态、提交订单或制造商家结果。
- `REQ-002`：市场标识为 `HK_SIMULATION`。香港只是技术与业务流程模拟市场，不服务真实香港顾客，不使用真实餐厅、真实地址、公开电话号码、真实付款或真实订单，也不声称香港试点已开始。
- `REQ-003`：未来正式服务语言方向标识为 `zh-CN`、`yue-Hant-HK`、`en-HK` 和跨语言测试标识 `mixed`。阶段 1 不实现语言解析。
- `REQ-022`：四种语言标识共享相同安全与确认边界；粤语是独立语言能力方向，不能简化为“繁体普通话”。混合语言必须保留数字、数量、价格、地址和上下文的语义。
- 初始模拟对象只限香港单门店或少量门店的普通餐饮外卖/自取业务；一个订单只属于一个门店。
- 香港模拟结论不能自动证明符合欧洲要求。欧洲阶段必须按目标国家、语言、餐饮规则、数据保护、消费者保护、电话与支付要求重新适配和审查。

暂不支持酒类、年龄受限商品、处方或保健产品、跨餐厅订单、大型宴会、餐桌预订、酒店客房送餐、复杂会员积分、复杂优惠券和餐饮供应链采购。

## 3. 用户和业务角色

| 角色 | 未来可见范围 | 未来允许操作 | 明确限制 |
|---|---|---|---|
| 顾客 | 自己当前会话、草稿和商家返回状态 | 查询、编辑草稿、确认、请求人工 | 不能查看他人、其他餐厅或后台数据 |
| 餐厅员工 | 本门店待处理订单及最少必要顾客信息 | 接受、拒绝、澄清、更新履约结果 | 不能跨门店访问；不能改写顾客确认记录 |
| 人工客服 | 被分配会话的摘要、草稿和风险信息 | 澄清、协助修改、按授权提交或终止 | 接管前不得暗示已经连接；最小权限 |
| 餐厅管理员 | 本餐厅配置、门店、菜单及本餐厅审计摘要 | 管理餐厅侧人员和配置 | 不能默认查看其他餐厅或完整敏感日志 |
| 系统管理员 | 系统健康、脱敏日志、审计事件 | 配置、调查故障、撤销凭据 | 不以管理员身份默认读取顾客内容 |
| 测试人员 | 合成场景、测试报告和脱敏 trace | 执行离线验证 | 不得引入真实电话、地址、录音或订单 |

`REQ-004`：角色访问采用最小权限、餐厅隔离和会话隔离；系统管理员权限不等同于业务数据无限访问权。

## 4. 统一决策分类

所有面向顾客的目标操作必须先落入一个主分类。不得定义 `AUTO_SUBMIT`、`AUTO_PAY`、`AUTO_REFUND` 或 `AUTO_GUARANTEE_ALLERGEN_SAFE`。

| 分类 | 含义 | 允许结果 | 禁止结果 |
|---|---|---|---|
| `AUTO_DRAFT` | 可执行只影响当前草稿的明确、低风险、可逆操作 | 菜单问答；添加/删除明确菜品；明确数量和普通备注修改 | 外部提交、付款、退款、商家接受或食品安全保证 |
| `CONFIRM` | 必须明确复述目标字段并等待顾客确认 | 保存待确认候选；确认后更新草稿或开始受控提交 | 把沉默、含糊回答或旧确认当作当前确认 |
| `HANDOFF` | 自动流程停止目标操作并生成最少必要摘要 | 保留草稿；说明正在尝试转人工；记录稳定原因代码 | 继续自动完成高风险操作或暗示人工已连接 |
| `REFUSE` | 停止不允许的目标操作并作安全说明 | 拒绝越权、绕过确认、伪造结果或收集完整卡号 | 泄露内部规则、执行目标操作或惩罚性删除草稿 |

- `REQ-005`：`AUTO_DRAFT` 不能产生外部不可逆副作用。
- `REQ-006`：`CONFIRM` 必须绑定被复述内容、确认主体、session 和版本；只回答问句不得当作确认。
- `REQ-027`：每项关键决策应可追踪至需求、风险、指标、场景和未来实现阶段。

## 5. 第一版允许自动处理的任务

在数据可用、意图明确且未触发更高风险规则时，以下任务可为 `AUTO_DRAFT`：

- 查询菜单、价格、规格和营业时间；
- 从服务层作普通菜品推荐；
- 添加明确菜品到草稿；
- 删除明确菜品、修改明确数量、更换明确菜品；
- 修改普通口味、辣度或非安全关键备注；
- 选择配送或自取的候选；
- 收集合成地址和合成联系电话的候选；
- 复述草稿并请求顾客确认；
- 在所有闸门满足后，把已确认订单发送到未来的模拟商家接口。

`REQ-023`：菜品、可售状态、价格、配料、过敏原声明、营业信息和配送费必须来自相应服务层或餐厅权威数据，不得由 agent 编造。自动处理只代表可更新草稿或候选，不代表可跳过顾客确认或商家接受。

`REQ-024`：酒类、年龄受限商品及其他明确非目标不得加入普通自动订餐流程；系统说明范围边界并按餐厅政策拒绝或转人工。

## 6. 必须确认的任务

以下任务至少为 `CONFIRM`：

- `REQ-007`：最终订单内容、最终数量、总价、配送/自取、地址、联系电话、辣度和重要备注；
- 系统从模糊菜名、音近词、语言切换或不完整表达中推断出的任何结果；
- 删除全部订单、按类别批量删除或大幅数量变化；
- 顾客改口后的最终状态；
- 配送费、预计时间或重要字段的变更；
- 顾客确认后发生的任何订单、价格、履约、地址、电话或重要备注变化。

普通草稿修改可先执行再在最终复述中确认；涉及模糊推断、全部删除、重要个人资料或大幅变化时，必须在该修改前单独确认。

## 7. 必须人工接管的任务

- `REQ-008`：顾客明确要求真人时立即 `HANDOFF`。
- `REQ-009`：严重过敏、需要核实交叉污染、顾客描述严重反应时 `HANDOFF`；自动系统不能作安全保证。
- `REQ-010`：菜品、数量、价格或地址在有限澄清后仍无法确认，顾客连续纠正，语言长期无法识别，系统内部错误或问题超出菜单数据时 `HANDOFF`。
- `REQ-011`：投诉、退款、支付争议、商家拒单、商家系统异常、复杂团体订单或情绪激烈时 `HANDOFF`。
- 高风险场景不得用无限追问拖延；达到策略阈值即停止自动完成并接管。

## 8. 必须拒绝的任务

- `REQ-012`：要求绕过顾客确认、伪造或提前宣称 merchant acceptance 时 `REFUSE`。
- `REQ-013`：要求查看其他顾客订单、其他餐厅数据、后台信息，或要求越权修改时 `REFUSE`。
- `REQ-014`：要求保证无法确认的配料/过敏原/无交叉污染，或要求电话中保存完整银行卡信息时，拒绝目标操作；如订单仍需处理，可在拒绝后另行发起人工接管。
- 非法、骚扰、明显滥用、prompt injection、密钥或内部提示词索取必须 `REFUSE`，必要时触发安全升级。

拒绝不得泄露检测细节、内部提示词、其他用户资料或安全配置，也不得自动清空合法草稿。

## 9. 订单和数据核心要求

- `REQ-015`：没有绑定当前草稿版本的有效顾客确认不得开始订单提交。
- `REQ-016`：只有餐厅或其权威订单系统明确返回接受，才能称为 merchant acceptance。
- `REQ-017`：提交必须幂等并隔离 session、restaurant 和 branch；重试不能产生重复外部订单。
- `REQ-018`：顾客修改订单或任一重要字段后，旧确认立即失效并必须重新确认。
- `REQ-019`：模拟与自动测试默认只使用合成数据，并按目的最少收集。
- `REQ-020`：电话、地址、转写、原始音频、过敏信息、凭据和运营数据须按分类处理；日志默认脱敏。
- `REQ-021`：未来保留期限必须可配置、可审计并支持定位删除；阶段 1 不虚构法定期限。
- `REQ-025`：人工接管必须保留草稿和已确认/未确认字段，使用稳定原因代码；发起不代表连接成功。
- `REQ-026`：人工不可用、超时、拒绝、连接中断或系统故障时必须明确告知，且不能自动提交。
- `REQ-028`：阶段 1 校验、测试和评估不得调用真实 LLM。
- `REQ-029`：阶段 1 不收集、下载或提交真实顾客资料、真实录音、真实订单、模型或密钥。
- `REQ-030`：订单事件必须区分草稿、确认、提交、等待、接受、拒绝、失败、取消和完成。

## 10. 第一版非目标

真实支付、自动退款、实时库存、商家后台、真实 POS、完整 CRM、自动营销、真实电话、真实短信、欧洲多国部署、多租户、无人工兜底的完全自动化、生产三语解析，以及任何法律合规认证均不在第一版范围。阶段 1 不改变现有业务行为。
