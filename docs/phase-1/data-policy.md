# 阶段 1 数据政策

本文是产品设计与风险控制规范，不构成法律意见，也不代表项目已取得香港、GDPR、欧洲或其他合规认证。任何真实试点前必须由适用地区的法律、隐私、安全和餐厅专业人员复核。

## 1. 分类体系

| 分类 | 定义 | 示例控制 |
|---|---|---|
| `PUBLIC` | 已获授权公开、泄露通常不会识别个人或暴露内部控制 | 可公开读取，仍需来源和完整性校验 |
| `INTERNAL` | 仅供项目或餐厅运营使用 | 身份认证、最小权限、禁止公开日志 |
| `PERSONAL` | 可直接或间接关联个人 | 目的限定、最少收集、传输/存储保护、访问审计 |
| `HIGH_SENSITIVITY` | 泄露或误用可能造成显著人身、隐私或歧视风险 | 默认不持久化或严格加密、细粒度授权、强化审计 |
| `SECRET` | 可授予系统或外部服务权限的凭据 | Secret Manager、禁止日志/仓库、轮换和撤销 |

分类是阶段 1 的工程控制模型，不替代适用法律中的法定分类。

## 2. 数据清单、目的和默认政策

| data_id | 数据类型 | 分类 | 允许目的 | 模拟阶段默认 |
|---|---|---|---|---|
| DATA-001 | 电话号码 | `PERSONAL` | 配送联系、经授权回拨 | 仅合成占位；仓库和日志不存完整值 |
| DATA-002 | 地址 | `PERSONAL` | 配送确认和范围核实 | 仅带“测试地址”的虚构地址；日志脱敏 |
| DATA-003 | 顾客姓名 | `PERSONAL` | 必要时识别订单/收餐人 | 默认不收集；场景仅用“测试顾客” |
| DATA-004 | 原始音频 | `HIGH_SENSITIVITY` | 经授权的语音处理或受控测试 | 默认不持久化，不进入仓库 |
| DATA-005 | 语音转写 | `HIGH_SENSITIVITY` | 理解当前会话、质量调查 | 只保留最少必要；完整文本不进普通日志 |
| DATA-006 | 订单内容 | `INTERNAL`；关联个人后为 `PERSONAL` | 建立草稿、提交和履约 | 合成订单；按 session/restaurant 隔离 |
| DATA-007 | 过敏信息 | `HIGH_SENSITIVITY` | 食品安全接管和餐厅核实 | 合成声明；不得用于画像或营销 |
| DATA-008 | 偏好/普通忌口 | `PERSONAL`（若可关联） | 当前订单推荐和备注 | 默认只用于当前目的，不默认为长期画像 |
| DATA-009 | 通话元数据 | `PERSONAL`（若可关联） | 路由、故障和计费核对 | 阶段 1 不生成真实电话元数据 |
| DATA-010 | 应用日志 | `INTERNAL` | 可用性、安全和故障诊断 | 不记录完整电话、地址、转写、音频或 key |
| DATA-011 | Trace | `INTERNAL`；含顾客内容时升级 | 决策追踪和离线评测 | 使用合成输入；字段级脱敏 |
| DATA-012 | 商家响应 | `INTERNAL`；关联订单后为 `PERSONAL` | 判断 pending/accepted/rejected | 仅模拟；保留权威来源和状态证据 |
| DATA-013 | 支付状态 | `HIGH_SENSITIVITY` | 未来只读状态核对/争议转接 | 阶段 1 不处理真实支付或卡数据 |
| DATA-014 | 测试数据 | `INTERNAL`，仅纯合成可在仓库长期保留 | 自动测试和回归 | 与任何未来生产数据物理/逻辑隔离 |
| DATA-015 | 模型评测数据 | `INTERNAL` | 质量和安全评测 | 仅合成；不得默认用真实会话扩充 |
| DATA-016 | 审计事件 | `INTERNAL`；含关联标识时为 `PERSONAL` | 追踪确认、提交、接受、接管和安全事件 | 记录结构化原因，不复制完整顾客原话 |
| DATA-017 | API key/令牌/签名密钥 | `SECRET` | 授权受控服务 | 禁止进入仓库、trace、摘要或对话 |
| DATA-018 | 菜单公开信息 | `PUBLIC` | 菜单问答与点单 | 必须来自服务层并校验版本 |
| DATA-019 | 内部风险规则 | `INTERNAL` | 分类、阻断和审计 | 对顾客只给必要说明，不泄露检测细节 |

## 3. 收集和使用规则

1. 默认使用合成数据；真实个人资料、真实录音和真实订单不得进入仓库。
2. 收集前必须有明确、具体的业务目的；只收集完成该目的所需字段。
3. 电话和地址在最终提交前必须复述确认；前端、日志和接管摘要默认掩码或分段显示。
4. 原始音频默认流式处理后丢弃。任何受控测试录音需要明确授权、用途、访问范围和清理计划，并与生产数据隔离。
5. Debug 模式也不得输出完整电话、地址、原始音频、完整转写、过敏详情、支付资料或凭据。
6. 完整银行卡号、安全码和磁道数据不得通过订餐对话收集或保存；支付争议直接 `HANDOFF`。
7. 过敏信息只用于当前食品安全目的，不得默认为营销、画像或模型训练材料。
8. 顾客内容和商家响应不得默认作为模型训练材料；任何未来二次使用需独立评估、适当告知/授权和退出机制。
9. 跨 restaurant、branch 或 session 数据访问必须被技术隔离和审计；管理员不拥有默认业务数据全读权限。
10. 测试数据和任何未来生产数据必须使用不同存储、凭据、访问角色和导出流程。

## 4. 保留和删除

### 阶段 1 已确定的原则

- 纯合成场景、schema 和不含真实个人资料的评测结果可长期保留用于回归。
- 真实个人资料、真实地址、真实电话、真实银行卡和真实录音不得进入仓库。
- 临时测试数据应在测试结束后清理；原始音频默认不持久化。
- 后续实现必须提供按数据类型、用途、环境和 jurisdiction 配置保留期限的入口，支持到期删除、法律保留例外标记、删除审计和定期复核。
- 删除请求必须能从授权标识定位相关数据及处理者副本，且不得依赖在日志中保存更多个人资料。

### 尚待确定

- 香港真实试点前：由产品、餐厅、隐私/安全负责人和香港法律专业人士确认各数据类型的具体期限、录音授权、通知文本、处理者安排、访问/更正/删除流程及事件通知方案。
- 欧洲阶段：按具体国家、数据角色、处理依据、电话录音规则、消费者/税务义务和欧盟要求重新确定；香港模拟期限不能直接复用。

本文不填写无来源的固定天数。保留期限应是经批准的配置和审计对象，而不是散落在代码中的常量。

## 5. 安全与事件处理

- 传输与存储保护、最小权限、密钥轮换、访问审计、备份删除和供应商处理者控制在后续阶段实现并验证。
- 检测到数据泄露时，先收集必要事实、遏制、评估对个人的伤害风险、由合格人员判断通知义务并记录决定；阶段 1 不预判具体法定义务。
- 日志注入、prompt injection 和越权请求不得改变数据访问策略。
- 任何脱敏/匿名化结论需验证重识别风险，简单遮盖不当然等于匿名数据。

## 6. 法律和专业复核清单

真实试点前至少确认：适用法律和数据角色；收集告知；处理目的/依据；录音同意与提示；敏感食品安全信息；供应商和跨境传输；访问、更正、删除与投诉流程；数据泄露响应；儿童/代理订餐；支付数据边界；餐厅过敏原责任；自动化决策和人工监督；各数据类型保留期限。
